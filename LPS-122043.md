# Identify New JS Browser Feature Usage Opportunities
* Language opportunities
* WebPlatform opportunities
* Polyfill-like removal like some methods of  `metal-dom`(We can get rid a lot of usages of metal-dom)

# Methodology:
* Research on CanIUse.com website features which are unsupported on IE11 but supported on Edge 80, Firefox 80, Chrome 80, Safari 13, Chrome for Android 85, iOS Safari 13.

You could see the results of this filter here:

https://caniuse.com/?compare=ie+11,edge+80,firefox+80,chrome+80,safari+13,ios_saf+13.0-13.1,android+81,and_chr+85&compareCats=HTML5,JS,JS%20API,Other,Security,SVG


* [crypto.getRandomValues()](https://caniuse.com/getrandomvalues)

It can be used to generate random values and can be an option for replacing Math.random() because it can generate most reliable random values at security point of view. See https://help.semmle.com/wiki/display/JS/Insecure+randomness for more information.

* [matches](https://caniuse.com/matchesselector)

It can replace easily metal-dom `match` method

Original implementation from metal-dom polyfill method:

https://github.com/metal/metal.js/blob/466812085dd29e37c90b835c64cb49555d76a294/packages/metal-dom/src/domNamed.js#L418..L435

Also, available on frontend-compabiliity-ie11 module:

https://github.com/liferay/liferay-portal/blob/a0397bf6adc1a9d6e60fb43579475241720f6526/modules/apps/frontend-compatibility/frontend-compatibility-ie/src/main/resources/META-INF/resources/closest.js#L15

* [keyboardEvent.key](https://caniuse.com/keyboardevent-key)

We already talked about it in the past on Clay’s repo: https://github.com/liferay/clay/issues/3430

Bringing some context, `event.keyCode` is a deprecated way to detect which keyboard key was pressed and We have other ways to do it. The most recommended is `event.key`:

On React use cases the mismatching key values are normalised but in vanilla use cases we need to treat those values on IE11 but since we’re dropping support to IE11 We could replace the usages with `event.key`. Also, We could create a keyboard mapping module for avoiding hard code strings to assert a Key, preventing the developers to provide a wrong key name.

* [`download` attribute](https://caniuse.com/download)

We have two occurrences on our codebase where people navigate to a pattern with a `download=true` ending for downloading a file. We could replace those occurrences with download attr.

See the occurrences here: 

https://gist.github.com/diegonvs/22a9e7065b3f948e6b487479e453d881

See more details about `download` attr here:

https://developers.google.com/web/updates/2011/08/Downloading-resources-in-HTML5-a-download

* [ `classList`](https://caniuse.com/classlist)

From https://caniuse.com/classlist, on IE11 we have limited support of `classList` causing some people using metal-dom’s  `addClasses` function. 

See the following result of  `git grep 'dom.addClasses'`:
https://gist.github.com/diegonvs/a9bb6953219e2104b83e446150d8e066

We can replace the usages with native classList functions.
